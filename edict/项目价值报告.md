# 三省六部 Edict 项目价值报告

> 历史说明：本文是早期审计版本，部分前端技术栈和能力状态已经过时。当前推荐阅读 [`项目价值报告V2.md`](项目价值报告V2.md)，并以根目录 [`../README.md`](../README.md) 与 [`../docs/getting-started.md`](../docs/getting-started.md) 为最新启动入口。

> 本报告基于当前项目源码与配置文件整理，重点区分“已经实现的能力”和“界面上存在但目前仍偏占位/兼容的能力”。结论尽量按实际代码说明，不做宣传式包装。

## 1. 一句话定义

Edict（三省六部总控台）不是一个普通聊天前端，也不只是一个“古风皮肤”的玩具。按当前代码看，它更准确的定位是：

**一个本地运行的 AI Agent 任务编排与审计系统：把用户的模糊指令转成可跟踪的多阶段任务，通过 AstrBot/OpenClaw 等 Agent 运行时执行，并把状态、进展、子任务、流转、最终奏折沉淀到数据库和仪表盘。**

它的核心价值不在“三省六部”的名称，而在这套机制：

- 有任务状态机，而不是一次性聊天。
- 有多角色分工，而不是一个 Agent 自说自话。
- 有状态流转、进展、todo、奏折，而不是只看最终日志。
- 有 Postgres/Redis/outbox/worker，任务过程可持久化、可恢复、可排查。
- 有一套强制 Agent 上报的 CLI 协议，把“AI 做了什么”落回看板。

## 2. 项目实际由哪些部分组成

### 2.1 后端服务

后端是 FastAPI 应用，入口在 `backend/app/main.py`。

它注册了这些主要 API：

- `/api/tasks`：原生任务 API。
- `/api/agents`：Agent 元信息和配置查询。
- `/api/events`：事件和 Redis Stream 信息查询。
- `/api/admin`：深度健康检查、pending 事件诊断、配置脱敏查询。
- `/ws`：WebSocket 事件推送接口。
- `compat.py` 中的一批 `/api/...` 兼容接口：当前前端主要还在使用这些扁平接口。

后端配置在 `backend/app/config.py`，支持：

- Postgres。
- Redis。
- OpenClaw adapter。
- AstrBot adapter。
- LLM 直调配置。
- 调度超时、重试、心跳、通知渠道等参数。

当前 `docker-compose.yml` 中实际默认使用的是 `DISPATCH_BACKEND=astrbot`，也就是通过 AstrBot 来调用模型/机器人能力。

### 2.2 数据库与事件系统

项目不是简单把状态写 JSON 文件。当前核心持久化依赖：

- Postgres：保存任务、事件、outbox 等结构化数据。
- Redis Streams：做任务事件总线。
- Redis Pub/Sub：给 WebSocket 做实时推送。

关键文件：

- `backend/app/models/task.py`：任务模型、状态枚举、合法流转、Agent 映射。
- `backend/app/services/task_service.py`：创建任务、状态流转、进度、todo、scheduler、奏折保存。
- `backend/app/services/event_bus.py`：Redis Streams 事件总线。
- `backend/app/workers/outbox_relay.py`：Transactional Outbox 投递端。
- `backend/app/workers/orchestrator_worker.py`：根据事件决定下一步派给谁。
- `backend/app/workers/dispatch_worker.py`：真正调用 AstrBot/OpenClaw Agent。

这说明项目的基础设计是偏“工作流系统/任务编排系统”，不是单页 demo。

### 2.3 前端仪表盘

前端是 React + Vite + TypeScript + Zustand。

关键文件：

- `frontend/src/App.tsx`：总控台 Tab 外壳。
- `frontend/src/store.ts`：前端状态、任务过滤、流程展示、模板定义。
- `frontend/src/api.ts`：前端 API client 和类型定义。
- `frontend/src/components/EdictBoard.tsx`：旨意看板，创建和查看任务。
- `frontend/src/components/SessionsPanel.tsx`：执行进度。
- `frontend/src/components/MonitorPanel.tsx`：省部调度。
- `frontend/src/components/MemorialPanel.tsx`：奏折阁。
- `frontend/src/components/TaskModal.tsx`：任务详情弹窗。
- `frontend/src/components/TemplatePanel.tsx`：旨库/模板下旨。
- `frontend/src/components/CourtDiscussion.tsx`、`CourtCeremony.tsx`：朝堂议政界面。
- `frontend/src/components/MorningPanel.tsx`：天下要闻界面。

当前前端主要是 5 秒轮询 `/api/live-status`，虽然后端已有 WebSocket，但前端主状态流还没有完全切到 WebSocket。

### 2.4 Agent 与 AstrBot 连接

当前项目有两套 Agent 调用适配器：

- `backend/app/adapters/astrbot.py`：通过 `POST /api/v1/chat` 调 AstrBot。
- `backend/app/adapters/openclaw.py`：通过 `openclaw agent` CLI 调 OpenClaw。

Docker 当前配置使用 AstrBot：

- 容器内 `ASTRBOT_API_URL=http://host.docker.internal:6185`。
- dispatcher 调用 AstrBot `/api/v1/chat`。
- 每个任务/官员组合会形成类似 `edict_{agent}_{task_id前8位}` 的 session。
- Agent 的角色设定来自 `agents/GLOBAL.md`、`agents/groups/*.md`、`agents/<agent>/SOUL.md`。

也就是说，Edict 自己不是模型服务，它是“调度层 + 看板 + 记账/审计层”，实际思考/生成由 AstrBot 或 OpenClaw 执行。

## 3. 已经实现的核心流程

### 3.1 标准任务流转

源码中定义的主流程是：

```text
Taizi -> Zhongshu -> Menxia -> Assigned -> Doing -> Review -> Done
```

对应到三省六部比喻：

| 状态 | 角色 | 实际作用 |
|---|---|---|
| Taizi | 太子 | 分拣任务，不直接执行 |
| Zhongshu | 中书省 | 起草方案、规划任务 |
| Menxia | 门下省 | 审核方案，可准奏或封驳 |
| Assigned | 尚书省 | 派发到六部或协调执行 |
| Doing | 六部 | 执行具体任务 |
| Review | 尚书省 | 汇总和复核执行结果 |
| Done | 回奏 | 任务完成，形成奏折 |

此外还有：

- `Blocked`：阻塞。
- `Cancelled`：取消。
- `Pending`、`PendingConfirm`：兼容/确认相关状态。
- `Next`：前端/兼容里用于部分待执行展示。

这套状态不是只给 UI 看的，`backend/app/models/task.py` 里有合法流转约束，非法跳步会被后端拒绝。

### 3.2 自动派发链路

实际链路是：

1. 用户在前端“下旨”或使用模板创建任务。
2. 后端 `TaskService.create_task()` 保存任务，并在同一事务写入 outbox event。
3. `outbox_relay` 把 outbox event 投递到 Redis Stream。
4. `orchestrator_worker` 消费事件，根据当前状态决定派发给哪个 agent。
5. `dispatch_worker` 消费派发事件，拼装任务上下文、角色 prompt、安全提醒、历史进展，然后调用 AstrBot/OpenClaw。
6. Agent 必须通过 `scripts/kanban_update_edict.py` 上报：状态、进度、todo、完成结果。
7. 看板轮询展示最新状态；后端也具备 WebSocket 推送基础。

这个设计的价值是：长任务不会只存在于某次聊天上下文里，任务过程能被保存、复盘、继续调度。

### 3.3 奏折/产物保存

项目现在已经不只是显示“完成了”。当前代码已经支持结构化 report：

- `Task.output` 保存摘要。
- `Task.meta["report"]` 保存结构化奏折。
- `scripts/kanban_update_edict.py done <id> <output_path> <summary>` 会上传：
  - 摘要。
  - 文件路径。
  - URL。
  - 文本文件正文预览。
  - 文件是否存在。
  - 文件大小。
  - 是否截断。

前端 `MemorialPanel.tsx` 和 `TaskModal.tsx` 会显示这些信息。

这点很关键：如果 Agent 做了调研、代码审查、方案设计，用户不用再去 AstrBot 日志里翻最终产物。

## 4. 这个项目真正有价值的地方

### 4.1 把 AI 从“聊天”变成“可审计任务”

普通聊天的问题是：

- 过程散在聊天记录里。
- 中间结论容易丢。
- 不知道任务卡在哪。
- 多 Agent 协作时更混乱。
- 最终结果如果没复制出来，就很难找。

Edict 解决的是“AI 工作过程管理”问题：

- 每个任务有 ID。
- 每个状态有责任角色。
- 每次流转有日志。
- 每段进展可上报。
- 每个 todo 可标记。
- 最终产物进入奏折。

这对“长任务、多步骤任务、需要复盘的任务”有实际价值。

### 4.2 强制 Agent 遵守流程，减少失控

`agents/GLOBAL.md` 里明确要求 Agent：

- 不许自己写代码直接调 API。
- 不许访问错误端口。
- 不许乱搜索任务数据。
- 必须用 `kanban_update_edict.py` 更新看板。
- 必须按状态机流转。
- 完成时必须写摘要。

`dispatch_worker.py` 也会把这些规则注入派发上下文。

这相当于给 Agent 加了一层“工作纪律”。它不能只在聊天里说“我完成了”，必须把完成状态写回系统。

### 4.3 适合管理长时间运行的本地 Agent 工作

当前 AstrBot API 调用超时配置、dispatch worker、scheduler 字段、stalled/retry/escalate 逻辑，都说明这个项目更适合“长任务看板”，而不是一次性问答。

适合的任务类型包括：

- 代码库调研。
- 方案设计。
- 多文件改造前的分析。
- 竞品/资料调研。
- 周报/总结类生成。
- 数据分析报告。
- 需要分阶段审核的复杂任务。

### 4.4 它能成为 AstrBot 的任务控制台

AstrBot 本身偏机器人/对话运行时。Edict 给它补的是：

- 任务创建入口。
- 多角色派发规则。
- 状态机。
- 进度面板。
- 完成报告归档。
- 失败/阻塞/重试可视化。

因此它的现实价值可以理解为：**给 AstrBot 加一个“项目经理/工单系统/审计看板”。**

## 5. 目前哪些地方还不能夸大

下面这些是根据当前源码看到的限制，使用时不要把它们当成已完整生产可用能力。

### 5.1 天下要闻目前偏占位

前端有 `MorningPanel.tsx`，也有订阅配置、企业微信 webhook UI。

但当前 `backend/app/api/compat.py` 中：

- `/api/morning-brief` 返回空 categories。
- `/api/morning-config` 返回空配置。
- `/api/morning-config` 保存接口只返回 `{ok: true}`。
- `/api/morning-brief/refresh` 也只返回 `{ok: true}`。

所以按当前代码，天下要闻界面还不能算完整新闻采集系统。

### 5.2 模型配置和派发渠道配置部分仍是 stub

前端有模型配置、派发渠道选择。

但 compat 后端里：

- `/api/set-model` 返回 `Model change recorded (stub)`。
- `/api/set-dispatch-channel` 返回 `{ok: true}`。

也就是说，当前 UI 上看起来能点，但未必真正改变底层 AstrBot 模型或持久化调度渠道。

### 5.3 WebSocket 后端存在，但前端主流程仍主要轮询

`backend/app/api/websocket.py` 已实现 `/ws` 和 `/ws/task/{task_id}`，通过 Redis Pub/Sub 推送事件。

但 `frontend/src/store.ts` 当前主加载逻辑仍是定时轮询 `loadAll()`，不是完全实时 WebSocket 架构。

### 5.4 可靠性依赖 Agent 遵守 CLI 协议

Edict 能展示进展和奏折的前提是：Agent 确实调用 `scripts/kanban_update_edict.py`。

如果 Agent：

- 只在 AstrBot 聊天里输出结果，
- 不调用 progress/state/done，
- 或调用错误脚本/错误端口，

看板就会卡住或缺少产出。

项目里已经加入了很多规则来约束 Agent，但本质上仍依赖 Agent 执行这些规则。

### 5.5 当前不适合直接暴露到公网生产环境

从源码和 compose 看，当前更像本地内网工具/开发环境：

- `docker-compose.yml` 里有开发数据库密码和 AstrBot API key 配置，真实使用应改为 `.env` 注入，避免提交/泄露。
- `DEBUG=true`。
- 没看到完整用户登录、权限控制、CSRF/鉴权体系。
- 后端管理接口和任务接口如果暴露公网会有风险。

建议仅在本机或可信内网使用。

### 5.6 数据库迁移与模型存在不一致风险

源码里有 `OutboxEvent`、`AuditLog`、`Todo` 等模型，但当前初始迁移 `migration/versions/001_initial.py` 和部分 ORM 模型并不完全一致：

- `outbox_events` 模型存在，worker 也依赖它，但初始迁移里没有创建该表。
- `audit_logs` 模型存在，但初始迁移里没有创建该表，当前也没看到它成为主流程审计入口。
- `todos` 迁移字段偏 `task_id/assignee/detail`，而 ORM `Todo` 模型字段偏 `trace_id/owner/assignee_agent/description`，两者不完全匹配。
- `/api/events` 能查询 Postgres `events` 表，但实际事件主链路主要是 outbox -> Redis Streams；当前阅读到的代码里没有确认所有 Redis 事件都会同步落 `events` 表。

这不影响“任务看板主流程”的定位判断，但说明：如果要把它当长期生产工具，需要先校准 Alembic 迁移、ORM 模型和实际 worker 依赖，避免新环境初始化后某些 worker 或审计接口不可用。

### 5.7 文档存在部分陈旧

`docs/astrbot-integration-guide.md` 里有些端点描述与当前 FastAPI 实际路由不完全一致。例如实际健康检查是 `/health`，任务 API 是 `/api/tasks`，而文档中出现过旧式 `/api/v1/...` 描述。

使用时应以源码和当前服务为准。

## 6. 怎么用它才“用到刀刃上”

### 6.1 不要把它当聊天框用

如果只是问一句“今天吃什么”“解释一个概念”，用普通聊天更快。

Edict 适合的是：

- 需要多步骤推进。
- 需要过程中可见。
- 需要最终产物归档。
- 需要不同角色审核。
- 需要以后复盘。

判断标准很简单：

> 如果这个任务值得有一个 ID、值得看进度、值得保存最终报告，就适合放进 Edict。

### 6.2 推荐使用场景一：代码库调研报告

适合下旨：

```text
通读某个模块源码，说明它负责什么、关键链路是什么、有哪些风险，最终输出 Markdown 报告到指定路径。
```

为什么适合：

- 中书省可以先规划调研范围。
- 门下省可以审核范围是否遗漏。
- 六部执行具体阅读和整理。
- 尚书省汇总成最终奏折。
- 奏折阁能看到摘要、文件位置、正文预览。

用法建议：

1. 在“旨意看板”下达明确任务。
2. 指定输出格式和路径，例如：`请输出到 docs/xxx-report.md`。
3. 看“执行进度”确认 Agent 有没有上报 progress。
4. 完成后去“奏折阁”查看结果和产物位置。

### 6.3 推荐使用场景二：复杂功能改造前的方案评审

适合下旨：

```text
分析当前项目中用户权限模块，提出最小改造方案，要求列出涉及文件、风险、验证步骤，不要直接改代码。
```

价值：

- 先让 AI 做方案，不直接动手。
- 门下省负责挑错、封驳或准奏。
- 方案进入奏折，后续可以让人类决定是否实施。

这比直接让 Agent 改代码更稳。

### 6.4 推荐使用场景三：长调研/竞品/资料整理

适合下旨：

```text
调研某个技术方案/竞品，输出：结论、证据、适用场景、不适用场景、建议。最终保存 Markdown。
```

价值：

- 可分阶段看进展。
- 最终报告不会丢在聊天里。
- 后续可归档为知识资产。

### 6.5 推荐使用场景四：日常工作流模板化

前端 `TemplatePanel.tsx` 和 `store.ts` 里已有模板能力，例如：

- 周报。
- 代码审查。
- API 设计。
- 竞品分析。
- 数据报告。
- 博客草稿。
- 部署检查。
- 邮件草稿。
- 站会同步。

建议你把最常用的 3 到 5 类任务固定成模板，减少每次下旨时的表达成本。

但要注意：模板只是创建任务的入口，真正质量仍取决于 Agent 执行和完成上报。

## 7. 推荐的实际工作方法

### 7.1 下旨要写清楚“产物”

不要只写：

```text
帮我调研一下这个项目。
```

建议写成：

```text
通读当前项目源码，输出一份 Markdown 报告到 项目价值报告.md。
报告必须包含：项目做什么、已实现能力、未完成/占位能力、真实价值、适合场景、不适合场景、如何使用。
不要编造源码里不存在的能力。
完成时通过 done 命令上传报告路径和摘要。
```

关键是明确：

- 读哪里。
- 产出什么。
- 保存到哪里。
- 哪些不能做。
- 完成后要回奏什么。

### 7.2 让 Agent 必须保存文件，而不是只回复文字

如果任务有复用价值，要求它保存到文件：

```text
最终输出到 docs/xxx.md，并在完成时把该路径作为 output_path 上报。
```

这样奏折阁才能展示路径和正文预览。

### 7.3 任务卡住时看三个地方

1. “省部调度”：看当前卡在哪个状态/部门。
2. “执行进度”：看 Agent 最近 progress 和 todo。
3. Docker logs：如果前端无进展，再看 dispatcher/orchestrator/backend 日志。

常见原因：

- Agent 没调用 `kanban_update_edict.py state`。
- Agent 调了错误脚本或错误端口。
- 状态跳步被后端拒绝。
- AstrBot API 无响应或模型调用失败。
- dispatcher 超时。

### 7.4 适合先“小任务试运行”再交给它大任务

每个新 Agent 或新任务类型，先用一个小任务验证：

- 能否收到派发。
- 能否上报 progress。
- 能否合法流转。
- 能否 done。
- 奏折阁能否看到产物。

确认链路通了，再跑复杂任务。

## 8. 最小可行使用步骤

### 8.1 启动系统

在项目根目录运行：

```bash
docker compose up -d --build
```

访问：

```text
前端：http://localhost:7899
后端：http://localhost:7898
```

### 8.2 确认 AstrBot 可用

当前 Docker 配置假设 AstrBot 在宿主机：

```text
http://localhost:6185
```

容器内通过：

```text
http://host.docker.internal:6185
```

如果 AstrBot 没开，任务可能创建成功，但派发执行会失败或无响应。

### 8.3 下达一个高质量测试旨意

建议测试任务：

```text
阅读 frontend/src/store.ts 和 backend/app/models/task.py，说明 Edict 的任务状态机如何工作。输出 Markdown 到 docs/state-machine-notes.md。完成时上报该文件路径和三句话摘要。
```

这个任务能验证：

- 创建任务。
- 多阶段流转。
- Agent 读取项目文件。
- 最终产物写文件。
- 奏折阁显示报告。

### 8.4 完成后检查

按顺序看：

1. 旨意看板：状态是否到 Done。
2. 执行进度：有没有 progress/todo。
3. 奏折阁：有没有摘要、路径、正文预览。
4. 文件系统：报告文件是否真实存在。

如果这四项都正常，说明 Edict + AstrBot 的核心闭环是通的。

## 9. 不适合用 Edict 的场景

不建议用它处理：

- 一句话问答。
- 临时闲聊。
- 不需要留痕的简单生成。
- 高风险生产操作，例如直接删库、改线上配置、发外部消息。
- 需要严格权限隔离和审计合规的公网多人系统。

这些场景要么普通聊天更快，要么当前项目安全边界还不够。

## 10. 如果要继续把它做成真正有生产力的工具，优先改什么

### 10.1 优先级一：把 stub 功能做实或从 UI 标明

建议优先处理：

- 模型配置真正持久化并影响 AstrBot 调用。
- 派发渠道配置真正保存并用于通知。
- 天下要闻采集/保存/推送要么做实，要么标记为未启用。
- 远程 skills 管理如果暂未实现，应在 UI 上明确说明。

否则用户会以为“点了就生效”，实际只是返回 ok。

### 10.2 优先级二：完善 Agent 完成产物规范

建议统一要求所有 Agent 完成时：

```bash
python3 E:/gitwork/edict/edict/scripts/kanban_update_edict.py done <任务ID> <产物路径或URL> "摘要"
```

并在 SOUL/GLOBAL 中明确：

- 调研类必须输出 Markdown。
- 代码类必须说明改了哪些文件和验证结果。
- 失败类必须说明阻塞原因和下一步建议。

### 10.3 优先级三：把前端接入 WebSocket

后端已经有 `/ws`，前端仍主要轮询。

如果接入 WebSocket，可以减少延迟，也能更像真正的“实时调度台”。

### 10.4 优先级四：安全与配置整理

建议：

- 不要把真实 AstrBot API key 写死在 `docker-compose.yml`。
- 改用 `.env`，并确保 `.env` 不提交。
- 增加最基本的访问鉴权。
- 管理接口不要暴露公网。
- 对外部 webhook、文件路径、Agent 输出继续保持边界。

### 10.5 优先级五：校准迁移、模型和审计链路

建议把数据库层做一次专项整理：

- 补齐 `outbox_events`、`audit_logs` 等模型对应的 Alembic 迁移。
- 统一 `todos` 表迁移与 ORM `Todo` 模型字段。
- 明确 `/api/events` 查询的 `events` 表是否作为正式事件归档；如果是，就让 Redis Stream 事件同步落库；如果不是，就在 UI/文档中降低它的审计承诺。
- 为新环境执行一次 `alembic upgrade head` 后的 worker 启动验证，确保 outbox、orchestrator、dispatcher 都不会因表缺失失败。

### 10.6 优先级六：清理过时文档和残留文案

当前仍存在一些历史痕迹：

- 文档中部分 API 路径陈旧。
- 部分模板选项还出现“飞书文档”。
- Feishu channel 后端仍保留，虽然可作为兼容，但 UI/默认应保持一致。

这些会影响用户理解系统真实能力。

## 11. 最终判断

这个项目现在的状态可以概括为：

**核心任务编排链路已经具备实际价值；部分外围功能仍是演示/占位；整体更适合作为本地 AstrBot Agent 工作台，而不是生产级 SaaS。**

它不是单纯的玩具，因为源码里已经有：

- 状态机。
- 持久化任务。
- Redis Streams 事件总线。
- transactional outbox。
- orchestrator/dispatcher worker。
- AstrBot/OpenClaw adapter。
- Agent 角色 prompt 注入。
- CLI 上报协议。
- 进度、todo、奏折归档。

但它也还没有完全成为成熟产品，因为：

- 有些 UI 功能仍是 stub。
- 前端实时性还没完全用上 WebSocket。
- 数据库迁移/审计链路还需要校准。
- 配置和安全还偏开发环境。
- Agent 执行纪律仍是成败关键。

所以最值得使用它的方式是：

> 把它当作“本地 AI Agent 工单系统”和“长任务审计看板”，专门承接那些需要分阶段推进、过程可见、最终产物可归档的任务。

如果你按这个定位使用，它就能从“好玩的三省六部皮肤”变成一个有用的 AI 工作流控制台。
