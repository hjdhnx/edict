# 三省六部 Edict 项目价值报告 V2

> 本报告基于当前项目实现重新梳理，重点说明 Edict 的设计理念、真实价值、适用方法与使用边界。它不是宣传稿，也不把本地工具夸大成公网生产级 SaaS；核心目标是帮助使用者理解：这个系统为什么存在、适合解决什么问题、应该如何用好。

## 1. 一句话定位

**Edict（三省六部总控台）是一个本地运行的 AI Agent 任务编排、治理与审计系统：把用户的模糊指令转成可跟踪、可审议、可执行、可归档的多阶段任务，并通过 AstrBot/OpenClaw 等 Agent 运行时完成实际工作。**

它不是普通聊天前端，也不只是一个“古风皮肤”的 UI。它真正想解决的是：

- AI 长任务过程不可见。
- 多 Agent 协作缺少规矩。
- 任务中间状态容易丢失。
- 最终产物散落在聊天记录或日志里。
- 出问题时无法判断卡在哪个环节。

Edict 的答案是：给 AI 工作流引入“制度”。每个任务有 ID，有阶段，有责任角色，有进展，有 todo，有流转日志，有最终奏折。

## 2. 设计理念：以古制御新技

“三省六部”不是装饰性命名，而是 Edict 的产品治理模型。

传统三省六部的价值在于分权、审议、执行和复核。Edict 借用这套结构，把 AI Agent 的工作从“一个模型直接回答”拆成更可控的流程：

| 阶段 | 角色 | 产品含义 |
|---|---|---|
| Taizi | 太子 | 分拣任务，判断方向和责任链路 |
| Zhongshu | 中书省 | 起草方案，拆解任务，形成计划 |
| Menxia | 门下省 | 审议方案，发现风险，必要时封驳 |
| Assigned | 尚书省 | 派发任务，决定执行部门 |
| Doing | 六部 | 执行具体工作，持续上报进展 |
| Review | 尚书省 | 汇总产物，复核结果 |
| Done | 回奏 | 完成任务，进入奏折归档 |

这套设计的核心不是“更有趣”，而是“更可控”：

- **规划和执行分离**：避免 Agent 一上来就动手。
- **审核和派发分离**：让风险在执行前暴露。
- **过程和结果分离**：既看最终产物，也看过程是否可靠。
- **聊天和工单分离**：把一次对话升级为可追踪任务。

一句话概括：**用古代政务的制衡结构，管理现代 AI Agent 的不确定性。**

## 3. 当前系统组成

### 3.1 后端调度与审计层

后端是 FastAPI 应用，围绕任务、事件、调度和兼容 API 构建。当前核心能力包括：

- 任务创建、状态流转、进度上报、todo 上报和奏折保存。
- Postgres 持久化任务、状态、元数据和报告信息。
- Redis Streams / PubSub 支撑任务事件和实时推送。
- Transactional Outbox 与 worker 链路用于可靠事件投递。
- orchestrator worker 根据状态决定下一步派发。
- dispatch worker 调用 AstrBot/OpenClaw 等外部 Agent 运行时。
- FastAPI `/api/live-status` 等接口向前端提供看板数据。
- 后端版本信息统一管理并展示到前端，例如 `V1.0.1 20260426`。

Edict 自己不是模型服务。它更像 AI Agent 的“项目经理 + 调度层 + 审计台”。

### 3.2 前端总控台

当前前端是 Vue 3 + Pinia + Ant Design Vue + Vite，目标是成为高信息密度的本地总控台。它已经具备：

- 白天/暗黑模式。
- PC 和移动端响应式布局。
- 紧凑型卡片、弹窗、表格和抽屉。
- 固定侧边栏和顶部栏，仅内容区滚动。
- 旨意看板、朝堂议政、省部调度、官员总览、模型配置、技能配置、执行进度、奏折阁、旨库、天下要闻等菜单。
- WebSocket 实时通道与轮询兜底。
- 版本号、品牌、favicon 等基础产品化元素。

它的重点不是做一个“大屏展示”，而是让用户快速回答这些问题：

- 任务现在在哪个阶段？
- 谁在处理？
- 最近有没有进展？
- todo 完成到哪里了？
- 是否阻塞或超时？
- 最终奏折和产物在哪里？

### 3.3 Agent 运行时与角色纪律

Edict 通过适配器接入 AstrBot/OpenClaw 等 Agent 运行时。实际推理、生成和工具调用由外部 Agent 完成，Edict 负责：

- 派发任务上下文。
- 注入三省六部角色规则。
- 要求 Agent 按协议上报状态、进度、todo 和完成结果。
- 将 Agent 的行为沉淀到看板和奏折阁。

这意味着 Edict 的质量不仅取决于后端和前端，也取决于 Agent 是否遵守工作纪律。系统通过 `agents/GLOBAL.md`、各角色 `SOUL.md`、dispatch prompt 和 `kanban_update_edict.py` 等方式约束 Agent：不能只在聊天里说“完成了”，必须把状态写回系统。

## 4. 当前真实能力

### 4.1 任务闭环

Edict 已经具备从“下旨”到“回奏”的主链路：

1. 用户在旨意看板或旨库模板中创建任务。
2. 后端保存任务并写入事件。
3. worker 根据状态派发给对应角色。
4. Agent 执行并上报进度、todo、状态变化。
5. 前端展示实时状态、活动流、耗时和阻塞信息。
6. 完成后形成奏折，保存摘要、产物路径、正文预览等信息。

这个闭环让长任务不再只存在于聊天上下文里，而是变成一个可被追踪、排查和复盘的工单。

### 4.2 执行可观测

系统围绕任务提供多维观察：

- 看板状态：任务处于哪个阶段。
- 流转日志：从哪个状态到哪个状态，原因是什么。
- 进度日志：Agent 具体做了什么。
- todo：任务拆解和完成比例。
- 耗时：活跃任务和归档奏折的时间成本。
- 调度信息：是否阻塞、是否需要重试、是否超时。
- 奏折：最终摘要和产物入口。

这些信息让用户不必猜“AI 到底有没有在工作”。

### 4.3 模型、技能与执行配置

当前系统提供模型配置、技能配置和远程技能管理界面，目标是让 Agent 能力可视化、可管理。它的价值不只是“展示配置”，而是把多 Agent 系统中最容易混乱的部分整理成面板：

- 当前有哪些官员/Agent。
- 每个 Agent 使用什么模型。
- 有哪些本地技能和远程技能。
- 技能来源、描述、更新和移除入口。
- 远程技能仓库的快速选择。

### 4.4 朝堂议政

朝堂议政是 Edict 的“审议界面”：用户可以围绕一个主题召集多个官员/Agent 表达意见，再由用户推进、裁决或形成后续任务。

它体现的是另一种产品价值：不是所有 AI 协作都必须直接进入执行流。有些问题更适合先讨论、辩驳、比较方案，再决定是否下旨。

### 4.5 天下要闻

天下要闻承担“外部信息输入”的角色。当前系统已经围绕分类、关键词、自定义信息源、RSS/网页采集和 webhook 推送做了实装方向：

- 用户可以配置订阅分类和关键词。
- 可添加自定义信息源。
- 可触发采集并在前端展示新闻卡片。
- 配置 webhook 后可按规则推送。

它让 Edict 不只是处理用户主动下达的任务，也可以成为日常信息输入和摘要入口。

## 5. 这个项目真正有价值的地方

### 5.1 把 AI 从“聊天”变成“工单”

普通聊天适合即时问答，但不适合管理长任务。Edict 给 AI 工作增加了工单属性：

- 每个任务有唯一 ID。
- 每个任务有阶段和责任角色。
- 每次流转可记录。
- 每段进展可上报。
- 每个 todo 可追踪。
- 每个结果可归档。

如果一个任务值得保存、值得复盘、值得知道它卡在哪里，就适合放进 Edict。

### 5.2 把多 Agent 协作变成制度化流程

多 Agent 系统常见问题是“热闹但不可控”：每个 Agent 都在说话，但用户很难知道谁负责什么、哪个结论被采纳、最终产物在哪里。

Edict 的三省六部机制让角色有边界：

- 太子负责分拣。
- 中书负责规划。
- 门下负责审核。
- 尚书负责派发和复核。
- 六部负责执行。

这种分工让协作变得像流程，而不是群聊。

### 5.3 把长任务变成可观察资产

对于代码库调研、方案设计、竞品分析、数据报告这类任务，用户真正需要的不是一句回答，而是一份可复用的产物。

Edict 的奏折阁让这些产物沉淀下来：

- 摘要可读。
- 路径可找。
- 正文可预览。
- 耗时可查看。
- 历史可回溯。

这让 AI 输出从“一次性回复”变成“项目知识资产”。

### 5.4 给 AstrBot/OpenClaw 补一个控制台

AstrBot/OpenClaw 更偏 Agent 运行时或执行入口。Edict 补的是上层管理能力：

- 谁来执行。
- 按什么流程执行。
- 执行到哪一步。
- 是否需要审议或复核。
- 最终产物如何归档。

因此 Edict 的现实定位可以理解为：**本地 AI Agent 工作台、工单系统和审计看板。**

## 6. 推荐使用方法

### 6.1 不要把它当普通聊天框

如果只是问一个概念、生成一句文案、临时闲聊，普通聊天更快。

Edict 更适合这些任务：

- 需要多步骤推进。
- 需要过程中可见。
- 需要不同角色审议。
- 需要最终产物保存。
- 需要之后复盘或复用。

判断标准：

> 如果这个任务值得有一个 ID、值得看进度、值得保存最终报告，就适合放进 Edict。

### 6.2 写好“旨意”的关键

高质量任务应包含五件事：

1. **范围**：读哪些文件、处理哪些模块、调研哪些对象。
2. **目标**：最终要解决什么问题。
3. **产物**：输出 Markdown、代码、表格、清单还是报告。
4. **路径**：产物保存到哪里。
5. **验证**：完成后如何证明结果可用。

推荐下旨格式：

```text
阅读 frontend/src/stores/dashboard.ts 和 backend/app/models/task.py，说明 Edict 的任务状态机如何工作。
输出 Markdown 到 docs/state-machine-notes.md。
报告必须包含：状态列表、合法流转、前端展示方式、可能的风险点。
完成时上报文件路径和三句话摘要。
```

### 6.3 推荐场景

#### 代码库调研

适合让 Agent 阅读指定模块，输出结构化报告：职责、关键链路、风险、改造建议。

#### 复杂功能方案

适合先让中书起草方案、门下审议风险，再决定是否执行，避免直接改代码带来返工。

#### 竞品/资料整理

适合长时间搜集信息、归纳证据、形成报告，并在奏折阁保存结果。

#### 日常工作流

适合周报、站会摘要、部署检查、API 设计、代码审查等可模板化任务。

#### 信息输入

天下要闻可作为外部信息源入口，把每日信息、关键词和 webhook 推送接入日常流程。

## 7. 不适合的场景

不建议用 Edict 处理：

- 一句话问答。
- 临时闲聊。
- 不需要留痕的简单生成。
- 需要毫秒级响应的交互。
- 高风险生产操作，例如删库、改线上配置、直接发布外部消息。
- 缺少权限和网络边界时的公网多人协作。

这些场景要么普通聊天更合适，要么当前系统边界还不应承担这类风险。

## 8. 边界与风险

### 8.1 本地和可信内网优先

Edict 当前更适合本地或可信内网使用。它连接模型、Agent、任务数据、webhook 和文件路径，这些都可能包含敏感信息。

如果要公开部署，需要补齐：

- 用户认证。
- 权限控制。
- 管理接口隔离。
- API key 和 webhook 脱敏。
- 访问日志与告警。
- 数据备份和恢复。
- 速率限制和任务配额。

### 8.2 可靠性依赖 Agent 上报纪律

看板展示进度的前提是 Agent 按协议上报。如果 Agent 没有调用进度、状态或 done 命令，前端就无法凭空知道真实执行情况。

因此使用 Edict 时，应持续维护 Agent 规则：

- 任务开始要上报。
- 阶段变化要合法流转。
- 长任务要定期 progress。
- todo 要及时更新。
- 完成必须保存产物并回奏。

### 8.3 外部集成需要安全边界

AstrBot、OpenClaw、webhook、自定义信息源、远程技能都属于外部集成点。使用时需要注意：

- 不要把真实密钥提交到仓库。
- 不要让 Agent 访问不可信路径。
- 不要将 webhook 用于敏感内容外发。
- 远程技能来源要可信。
- 自定义信息源要防止异常内容污染摘要。

## 9. 后续演进建议

### 9.1 强化权限和生产安全

如果 Edict 要从本地工具走向团队工具，优先补齐认证、角色权限、接口隔离、密钥管理和审计日志。

### 9.2 强化产物规范

建议为不同任务类型定义完成规范：

- 调研类必须输出 Markdown 报告。
- 代码类必须列出改动文件、测试结果和风险。
- 数据类必须保存数据来源、处理过程和结论。
- 失败类必须说明阻塞原因和下一步动作。

### 9.3 强化模板体系

将高频工作沉淀为模板，减少每次下旨的表达成本，也能提高 Agent 执行一致性。

### 9.4 强化审计和迁移一致性

长期使用时，应持续校准数据库迁移、ORM 模型、事件表、outbox 和 worker 依赖，保证新环境初始化后链路完整。

### 9.5 强化实时体验

继续优化 WebSocket、任务活动流和局部刷新，让前端更像真正的实时调度台。

## 10. 最终判断

Edict 当前最准确的定位是：

**一个面向本地 AI Agent 的工单系统、任务治理层和审计看板。**

它的价值不在于替代聊天机器人，而在于管理聊天机器人不擅长的部分：长任务、多阶段、多角色、可见过程、可复盘产物。

如果把它当普通聊天框，它会显得重；如果把它用于代码调研、方案设计、报告生成、复杂任务拆解和多 Agent 协作，它的三省六部结构就能发挥价值。

最终，Edict 想提供的是一种更稳的 AI 工作方式：

> 不是让 AI 更热闹地回答，而是让 AI 更有规矩地工作。
